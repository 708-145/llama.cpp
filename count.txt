cmake --build build --config Release -j8 --target llama-cli

Test with Tiny-Moe.Q5_K.gguf
wget https://huggingface.co/mradermacher/Tiny-Moe-GGUF/resolve/main/Tiny-Moe.Q5_K_M.gguf -O Tiny-Moe.Q5_K.gguf

Inference:
./build/bin/llama-cli -p "What is 3+4?" -n 38 -m Tiny-Moe.Q5_K.gguf

static information (from model load):
llama_model_loader: - type q5_K: 73 tensors
llama_model_loader: - type q6_K: 13 tensors

Original dynamic information (counting src0 if quantized in mul_mat):
inference-time quantization type usage counts (from ggml): 
q5_K: 11088 (291.79 per token for 38 tokens)
q6_K: 1848 (48.63 per token for 38 tokens)

Original bug observation:
Static and dynamic observations (when counting src0) didn't agree with the expectation of counting weight tensors. The calculation assumed 38 tokens were processed. The high counts for src0 suggested activations were being counted multiple times or many quantized activation tensors were involved.

**Current Investigation & Findings (Counting src1 in mul_mat):**

Goal: To count the usage of *original quantized weight tensor types* within `GGML_OP_MUL_MAT`.

Change Made:
- Modified `ggml-cpu/ggml-cpu.c` in `ggml_compute_forward_mul_mat` to attempt to count `src1->type` (weights) if `src1` is quantized, instead of `src0->type` (activations).
  The line `if (ggml_is_quantized(src0->type)) { ggml_quant_stats_increment(src0->type); }`
  was changed to `if (ggml_is_quantized(src1->type)) { ggml_quant_stats_increment(src1->type); }`.

Test Results with new code (counting src1):
- When running `./build/bin/llama-cli -p "What is 3+4?" -n 38 -m Tiny-Moe.Q5_K.gguf`, the output for dynamic counts is:
  `inference-time quantization type usage counts (from ggml): (no quantized operations detected or stats not collected)`
- This means Q5_K and Q6_K counts are zero.
- Debug prints added temporarily to `ggml_compute_forward_mul_mat` confirmed that `src1->type` (the weight tensor operand) is consistently `f32` for all `GGML_OP_MUL_MAT` calls during the test run. `src0->type` (activations) showed various types, including `q5_K`, `q6_K`, `f16`, and `f32`.

Conclusion:
- The reason for zero counts for Q5_K/Q6_K when attempting to count `src1->type` inside `ggml_compute_forward_mul_mat` is that the weight tensors (`src1`) are already dequantized to F32 by the time this specific function is called in the execution graph for this model and command.
- The original "bug" where static and dynamic counts didn't agree was likely a misinterpretation: the dynamic counts were reflecting quantized *activations* (`src0`), not *weights* (`src1`).
- To accurately count how many times the *original* Q5_K/Q6_K weight tensors from the model file participate in `GGML_OP_MUL_MAT` (even if they are dequantized to F32 immediately before the actual `mul_mat` operation in the graph), the counting mechanism needs to:
    1. Identify `GGML_OP_MUL_MAT` operations.
    2. For `src1` of these operations, trace back to its original tensor type if `src1` is an F32 tensor that resulted from a dequantization operation (e.g., `GGML_OP_CPY` from a quantized type to F32).
    3. Or, instrument graph creation/traversal logic to use the helper `ggml_increment_compute_quant_stats_for_mul_mat(src0->type, src1->type)` with the types of the tensors as defined in the graph node for `GGML_OP_MUL_MAT`, before any implicit dequantization for computation occurs.

The current simple instrumentation counts what the `ggml_compute_forward_mul_mat` function *directly receives*. If `src1` is F32 there, it's counted as F32.
